## 4. 사용자 질문에 대한 답 — "격리≠in-model = config 문제였나?"

**부분적으로 맞다.** 12x 격차 = **config 성분**(격리가 저품질 컴파일) + **context 성분**(cross-op fusion/overlap 상실, composition sweep에서 "compute 100%가 weight-load 그림자에 숨음" 확인). 2026.2.0에선 격리를 저품질로만 컴파일 가능 → **두 성분을 분리 못 했다.** 2026.3.0은 격리를 near-production 품질로 **컴파일+실행**하게 해줌 → 처음으로 **분리 실험 가능**:

> 격리 커널/번들을 ForLlmModelComputeBound+scheduler_beam_search+use_attention_kernel로 컴파일 → TpModule.run으로 실측 → in-model 실측과 비교. 잔차 = (77-knob 잔여 config) + (context). 작으면 bottom-up 부활, 크면 context가 진짜 벽 확정.

이 실험은 2026.2.0에선 **원천 불가**였다. = 컴파일러 업데이트의 실질 가치.

### 7.1 환경·API (2026.2.0과 다름, 하드-won)
- fresh venv `/home/furiosa/venv3030` (python3.10). `pip install furiosa-torch==2026.3.0` → torch 2.10.0 딸려옴. furiosa-native-llm-common은 안 깔아도 됨(격리 실험엔 불필요). **PyPI 기본 인덱스에 furiosa 있음**(pip config 비어도 설치됨). RNGD device 정상(2026.2.x 드라이버와 충돌 없음, `furiosa:0`~`:7` 8개).

(2026.2.0의 `furiosa.torch.native_device`, `furiosa.torch.compiler.Config`는 없어짐.)

(4) 타이밍: `cm()`이 **async** → wall-clock은 바닥값만 잼. **`stream.synchronize()` 배치**(ITERS 호출 큐잉 후 1회 sync, /ITERS)로 정확한 device time. (5) 종료 시 device 정리 **teardown 크래시** → `os._exit(0)`로 우회. (6) stdout 파이프 버퍼링+os._exit → 출력 유실, `python -u` 또는 flush.

### 7.3 핵심 결과 (1B, TP=8, 레이어당 Tokenwise, device time us)

| seq | 개별 matmul 합(A) | 번들(B) | in-model(2026.2.0 프로파일, @1.1GHz) | A/in-model | B/in-model |
|----|----|----|----|----|----|
| 512 | 720.6 | 450.9 | 752 | **0.96x** | 0.60x |
| 1024 | 1096.9 | 776.8 | 1299 | **0.84x** | 0.60x |

- 격리 Tokenwise 번들 ForLlm: 128→201, 512→451, 1024→777, 2048→1358us (거의 선형, 안 터짐). NoConstraint: 128→189, 512→420, 1024→659.
- **개별 matmul 합 ≈ in-model(0.84~0.96x). 예전 "합 > 전체 12x"가 재현 안 됨.** 개별 op을 따로 컴파일해도 고품질이면 뻥튀기 안 됨. 개별합/번들=1.4~1.6x(번들이 op끼리 overlap해서 더 빠름).
- **config knob(NoConstraint vs ForLlm)은 이 memory-bound 영역에선 latency 거의 안 바꿈(~10%, ForLlm이 오히려 살짝 느림).** 즉 벽을 뚫은 건 특정 knob이 아니라 **runnable 고품질 compile 경로 자체**(2026.3.0).

### 7.4 해석 (07-16 결론 수정)
- **예전 12x/347ms = compute-bound에서 저품질 World-1 컴파일(5% MFU) 아티팩트였다** (고품질 넣으니 개별합이 in-model과 맞음 = 사용자 가설 맞음). "NPU latency=컴파일 문맥 속성"은 여전히 참이나, **그 문맥을 격리에도 줄 수 있게 됨** → 격리↔in-model 격차가 config로 대부분 설명됨.
- ⚠️ 한계(정직): (1) 검증 영역이 **memory-bound(S≤1024)** — 12x가 살던 compute-bound(큰 S/8B) in-model 대조는 아직. (2) 격리 번들이 in-model과 **op 완전 동일 아님**(RoPE·RMSNorm×2·residual·kv-scatter 빠짐) + in-model은 계측빌드·다른 컴파일러버전(2026.2.0) → in-model이 다소 부풀려짐. 그래서 0.6~0.96x는 순수 비교 아니고 "같은 자릿수" 수준. (3) config 23/100 knob만 노출(partial_sum_policy 등 미노출).

1. **op 동일 정합**: 격리 번들에 RoPE·RMSNorm·residual·kv-scatter 넣어 in-model supertask와 op-identical하게 → 깨끗한 격리↔in-model 비교(순수 context 잔차 측정).
2. **compute-bound 대조**: 큰 S 또는 8B에서 격리 고품질 vs in-model(8B in-model 필요) — 12x가 정말 닫히는지 결정 검증.
3. **n-gram 조합 법칙**: 격리 개별→2/3-gram→번들 흡수율을 실측(2026.2.0 composition_sweep을 이제 실행값으로) → 조합 예측기.

## 9. ★n-gram 조합 법칙 실측 — bottom-up 예측 성립 (device-cycle)★

목표(사용자 확정): **ground truth = 실제 LLM forward의 in-model Tokenwise latency, 재료 = 개별 격리 측정.** 개별→조합→in-model 예측·검증.

### 9.1 측정 방식이 결정적 — host wall-clock은 안 됨, device-cycle이어야
- host perf_counter는 **op당 ~40us 고정 바닥**(dispatch/launch 오버헤드; resid·rope가 seq 무관하게 ~40us). 개별 op 13개 합하면 이 바닥이 13× 뻥튀기 → 작은 seq 과대예측.
- **해결: 프로파일러 device cycle.** `from furiosa.torch.profiler import Profiler; with Profiler(profile_path=pp): run×N`. env `TUC_PROFILE_LEVEL=info RUST_LOG=info,span::tuc=info`. trace의 `name=="Task"` 스팬 args에 `cycle_actual`/`begin_cycle`/`end_cycle`/`chip/cluster/pe` = **device cycle**. TP=8 = 2 cluster(pe0-3, pe4-7). warmup 뺀 min 취함. **in-model GT도 cycle 단위 → cycle 대 cycle 직접 비교(클럭 변환 불필요).**
- 검증: device cycle이 host 바닥 제거 확인 — resid@512 device 12.3K cyc(~8us) vs host 41us; gate@512 252K cyc(~168us)≈host 160us(큰 op은 원래 일치). **클럭 ≈1.5GHz**(gate 252K/160us). ⚠️ 예전 메모리 "1.1GHz"는 부정확했을 수 있음(단 GT를 cycle로 비교하면 무관).

### 9.2 결과 — 조합 법칙 (개별 device-cycle 합 vs in-model Tokenwise cycle)

| seq | in-model GT(cyc) | naive_mm/GT | naive_all/GT | (참고 host naive_all) |
|----|----|----|----|----|
| 128 | 368,000 | 1.28x | 1.42x | (2.09x) |
| 512 | 827,000 | **1.19x** | 1.28x | (1.36x) |
| 1024 | 1,429,000 | **1.18x** | 1.27x | (1.02x) |

- **host의 seq-요동(0.82~2.09x)이 device-cycle에선 ~1.2x로 안정.** = 측정 정밀화가 조합법칙을 드러냄.
- **★조합 법칙: in-model Tokenwise ≈ Σ(개별 op device-cycle) ÷ 1.2.★** matmul 합만 쓰면 seq≥512에서 1.18~1.19로 매우 안정. 이 1.2배 = supertask 내 overlap/흡수(weight-load 뒤 compute 은닉).
- 비matmul(RMSNorm·RoPE·resid, 각 5~25K cyc)은 in-model서 거의 완전 흡수 → naive_all이 오히려 더 과대예측.
- **= 개별 측정만으로 in-model ~20% 안에서 예측 = bottom-up 성립.** 사용자 원래 n-gram/absorption 아이디어가 정밀 측정 위에서 성립.
- ⚠️ caveat: GT=2026.2.0 계측빌드, 재료=2026.3.0 → 버전차 systematic offset이 1.2배에 섞였을 수 있음. **같은 2026.3.0에서 GT 새로 뜨면 1.2가 순수 overlap인지 분리됨** = 다음 단계.

## 11. ★진짜 n-gram 방식 유효성 실증 (device-cycle, telescoping)★

목표 재확인(사용자): **LLM을 커널 n-gram 시퀀스로 보고, n-gram 묶음 측정으로 ML 학습→전체 추론시간 예측.** 그 전제 = "n-gram 묶음을 재면 그 안에 op끼리 overlap/문맥이 담긴다"가 참인지. (9절까지는 unigram=개별합이었고, 이건 진짜 n-gram.)

방법: chain [q,k,v,o,gate,up,down]. 연속 2/3개 묶음(window)을 device-cycle 측정 → telescoping(공유 op/2gram 빼며 이어붙임)으로 전체 번들 예측. 예측 대상=full 7-op 번들(같은버전 clean).

| seq | unigram 합/full | 2-gram tele/full | 3-gram tele/full |
|----|----|----|----|
| 512 | 1.44x | **1.06x** | 0.69x* |
| 1024 | 1.27x | **1.03x** | 1.02x |

*512의 3-gram qkv/kvo가 0으로 측정실패(노이즈)→무효. 2-gram은 둘다 깨끗.

- **unigram 합은 27~44% 과대예측, 2-gram telescope는 3~6% 안.** = op 묶음(n-gram) 측정이 overlap을 포착 → unigram이 놓친 것. **n-gram 방식 유효 실증.**
- telescoping 공식: `tele2 = W2(0)+Σ_{i≥1}(W2(i)−U(op_i))`, overlap=0이면 unigram합으로 환원(검산됨).
- 코드 `ngram_telescope_30.py`. 다음: n-gram을 (op구성·shape·문맥→latency) 데이터로 ML 학습 = 4단계(진짜 목표).
