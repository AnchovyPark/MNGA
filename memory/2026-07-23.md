# RNGD LLM 연구 — 2026-07-23 세션 기록

## 0. 요약

**컴파일러 2026.3.0이 "config 벽"을 크게 낮춘다는 걸 stub 분석으로 확인.** 사용자 질문("격리가 in-model과 다른 게 config 문제 아니었어? 컴파일러 업데이트로 풀릴까?")에 대한 답:
**부분적으로 yes. 2026.3.0에서 (1) 실행 가능한 World-1 경로가 production knob을 훨씬 많이 받고 (2) 결과를 standalone 실행까지 한다. 그래서 예전엔 불가능했던 "config vs context 분리 실험"이 처음으로 실행 가능해졌다.** 단, full 100-knob 완전 노출은 아니고(23/100), World-2 full-config는 여전히 실행 불가. 하드웨어 미검증(stub 기준).

- 현재 작업 venv(`furiosa-llm-getting-started/.venv`): **2026.2.0** (rev 9f92da06, 2026-04-28 빌드).
- `tmp-swj-test/.venv`: **2026.2.1** — World-1 표면이 2026.2.0과 **완전 동일**(우리 문제엔 무의미).
- pip 인덱스 최신: **2026.3.0** (TCL 커스텀 커널 SDK 버전과 동일). 사용자가 다른 세션에서 이걸 설치 중.
- device 충돌 없이 **2026.3.0 wheel을 설치 없이 다운로드 → `.pyi` stub만 분석**으로 표면 확인 (RNGD 안 건드림).

## 2. 2026.2.0 → 2026.3.0 World-1 변화 (핵심)

| 항목 | 2026.2.0 | 2026.3.0 |
|---|---|---|
| `compile()` 반환 | `ir.Edf` | **`Runnable`** (`compile(ep, compiler_config, dram_shapes)`) |
| World-1 `CompilerConfig` knob 수 | 12 | **24** |
| tactic_hint enum | Default/ForLlmModelDecode/Prefill/PrefillLongSequence/... | Default/NoConstraint/**ForLlmModelComputeBound**/ForLlmModelIOBound/ForLlmModelComputeIOBalanced/... |
| 실행 | ir.Edf→EdfModule | **`TpModule.from_runnable(r, inputs).run()`** + `DfgExecutor.execute(inputs)` |
| World-2 full-config 실행 | 불가(CompiledGraph run 없음) | **여전히 불가** (CompiledGraph = is_edf/is_pre_command_gen/serialize/deserialize만) |

**새로 노출된 production knob** (12→24 증분): `scheduler_beam_search`, `expected_total_beam_states`, `use_attention_kernel`, `weight_sharding_threshold_in_bytes`, `tensor_unit_bridge_threshold_in_page`, `local_population_threshold`, `padding_policy(_small)`, `sparsify_moe`, `allow_reduce_by_ve_cluster_chip_reduce(+base_population)`, `use_dma_bridges_only`, `progress_mode`.

## 3. 정량 커버리지 — production 100 knob 중 23개 노출

`create_llm_compiler_config_with_layer_range("llama",...)`로 production dict 생성 → top-level **100 키**. 2026.3.0 World-1 `CompilerConfig`의 24 knob과 교집합 = **23개** 노출.

- **노출됨(23)**: tactic_hint(+ForLlmModelComputeBound), tactic_sorting_policy, scheduler_beam_search, expected_total_beam_states, use_attention_kernel, weight_sharding_threshold_in_bytes, tensor_unit_bridge_threshold_in_page, local_population_threshold, padding_policy, sparsify_moe, VE cluster reduce 2종, use_dma_bridges_only, progress_mode, + 기존 type-casting/tactic-pruning류.
- **아직 안 됨(77) 중 latency 핵심**: `partial_sum_policy`(예전에 콕 집어 막혔던 것 — 여전히 막힘), `tactic_context_config`, `schedule_context_config`, `hide_qk_einsum_under_ve`, `arithmetic_qk_masking`, `enable_einsum_fusion`, `apply_adaptive_einsum_by_pattern`, `use_aligned_repartition2`, `lowering_mode`, `enable_chip_innermost`.

→ 결론: **가장 영향 큰 레버는 열렸지만 full fidelity(byte-identical)는 아님.** 남은 77 knob(특히 einsum-fusion/partial-sum/context 계열)은 여전히 실행 불가한 World-2 경로에만.

## 4. 사용자 질문에 대한 답 — "격리≠in-model = config 문제였나?"

**부분적으로 맞다.** 12x 격차 = **config 성분**(격리가 저품질 컴파일) + **context 성분**(cross-op fusion/overlap 상실, composition sweep에서 "compute 100%가 weight-load 그림자에 숨음" 확인). 2026.2.0에선 격리를 저품질로만 컴파일 가능 → **두 성분을 분리 못 했다.** 2026.3.0은 격리를 near-production 품질로 **컴파일+실행**하게 해줌 → 처음으로 **분리 실험 가능**:

> 격리 커널/번들을 ForLlmModelComputeBound+scheduler_beam_search+use_attention_kernel로 컴파일 → TpModule.run으로 실측 → in-model 실측과 비교. 잔차 = (77-knob 잔여 config) + (context). 작으면 bottom-up 부활, 크면 context가 진짜 벽 확정.

이 실험은 2026.2.0에선 **원천 불가**였다. = 컴파일러 업데이트의 실질 가치.
