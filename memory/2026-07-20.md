# RNGD LLM 연구 — 2026-07-20 세션 기록

## 0. 요약

07-19에 시작한 8B 일반화 검증 완료. **핵심 결과: 컴파일러 예상치 보정계수(k_type)가 1B·8B에서 동일 = 사실상 보편 상수.** 그리고 다음 방향(supertask ML 예측기)을 확정했다.

## 1. 8B 일반화 확인 — k_type이 모델 크기에 무관

`meta-llama/Llama-3.1-8B-Instruct`(raw HF, gated, 접근권 있음)를 다운받아 dump+profile 빌드 → 1B와 똑같이 in-model 예상 vs 실측 대조.

```text
8B: seq   TW예상      TW실측     k_TW  | attn예상  attn실측  k_Attn
    128   540479     909090    1.68x  | 16950    42453    2.50x
    256   723373     1271140   1.76x  | 29572    95155    3.22x
    512   1275556    2288494   1.79x  | 77477    167663   2.16x
    1024  2402099    4286203   1.78x  | 218898   472953   2.16x

              8B 평균        1B
    k_TW      1.75     ==    1.75
    k_Attn    2.51     ==    2.5
```

- **k_TW = 1.75 (1B) = 1.75 (8B). k_Attn = 2.5 (1B) = 2.51 (8B).** 모델 크기 4배 차이인데 보정계수 동일.
- **결론: 실제 latency = 컴파일러 dump 예상 × k_type, k_type은 seq·모델크기 무관.** → 아무 Llama 모델/config든 **컴파일만 하면(안 돌려도) 컴파일러 예상치 × k_type = 실제 latency 예측.** 계수는 한 모델서 calibrate하면 전이됨. **= bottom-up 예측이 성립하는 지점.**
- ⚠️ furiosa-ai/Llama-8B는 raw 안 줌(furiosa params-*/prebuilt만) → 재빌드 불가. raw는 meta-llama org. latency는 weight 값 아니라 shape+config로 정해지므로 raw 빌드 ≈ furiosa official.

## 2. 컴파일러 예상치 = 정확히 뭔가 (기록)

`{tag}.summary/summary.json`은 **supertask 하나당 하나**. 안에 두 레벨:
- **supertask 레벨**: `total_cycle`(전체 예상 latency, ×k 한 것), `io_cycle`, `computation_cycle`.
- **op(kernel) 레벨**: `operator_cycle` = {op이름: cycle} — supertask 안 각 op별 cycle (역할명: attn_Q_proj, mlp_gate_proj, matmul_qk, softmax, ""=overhead).
- **★op별 합 ≠ total★**: 컴파일러가 op들을 겹쳐(overlap) 일부 숨김 → total < 합. supertask latency는 커널 단순합이 아니라 자기 숫자. **이 차이(overlap)가 컴파일러 최적화.** (예: 8B O+MLP seq512: op합 1.62M vs total 1.10M.)

## 3. 다음 방향 (주현 확정) — Supertask ML 예측기

목표: **Kernel → Supertask → Real**. 격리 커널 측정은 죽었으니(컴파일러가 격리↔in-LLM 다르게 동작, 1:1 매칭 불가), **supertask를 원자 단위로** 삼고:

- **각 supertask를 "문맥"으로 featurize**: 안에 든 커널들(op 구성) + 각 커널 shape + 앞뒤 문맥.
- **ML 학습** (문맥 → latency), **안 본 supertask 예측**.
- = 주현 원래 n-gram/context 아이디어를 supertask 단위로 옮긴 것.

**데이터·feature 다 확보됨:**
- 학습 타겟 = supertask 실측 latency (profiled 빌드 실행, 모델당 ~164개).
- feature: op 구성(operator_cycle op 이름), 커널 shape(아티팩트 tensor + bucket), 컴파일러 예상치(total/io/compute/overlap = 초강력 feature).
- 설계 선택: 컴파일러 예상치를 feature에 넣냐(정확·근데 컴파일 필요) vs shape/구성만(싸고 야심참). 둘 다 넣으면 예상치 base + 구성/shape 보정.
