# RNGD LLM 연구 — 2026-07-16 세션 기록

## 0. 오늘의 큰 방향

07-03 이후의 큰 전환점. 그동안 "격리 커널 프로파일 → 조합(n-gram) → 실제 LLM latency 예측"(bottom-up)을 밀어왔는데, 이번 세션에서 **이 방향이 이 NPU에서 근본적으로 막혀 있음**을 규명했다.

핵심 질문 두 개:

> 우리가 짠 커널(혹은 2~3개 묶은 n-gram)을 **실제 LLM과 같은 production 품질로 컴파일**해서 측정할 수 있는가?
> 안 된다면, 왜 안 되고 무엇이 남는가?

결론부터: **격리 커널/n-gram을 production 품질로 컴파일하는 길은 SDK에 전부 막혀 있다.** 그리고 그 이유가 곧 발견이다 — NPU에서 커널 latency는 커널 고유 속성이 아니라 **"어떻게 컴파일됐냐(컴파일 문맥)"의 속성**이다.

## 1. 문제의 실체 — 부분의 합이 전체의 12배

Llama-3.1-8B, prefill, S=2048 기준:

```text
진짜 추론 (prefill)                 : 28.1 ms
그 추론에 들어가는 행렬곱 시간만 더한 값 : 347 ms   (격리 측정, World-1)
→ 부분(행렬곱)의 합이 전체 추론의 12.4배
```

같은 gate 행렬곱이:

```text
격리 컴파일        : 2457 us  (~5% MFU)
실모델 안 융합+튜닝 : ~130 us  (~90% MFU)
→ 같은 연산인데 19배 차이
```

즉 combiner(n-gram이든 matmul-anchor든)를 아무리 잘 만들어도 소용없다. **재료(격리 측정값) 자체가 12~19배 뻥튀기**되어 있기 때문. 이게 그동안 "cross-graph 2배 gap"이라고 부르던 것의 정체이고, 실제로는 S=2048에서 12배다.

## 2. 왜 뻥튀기되나 — 두 개의 분리된 컴파일러 표면

같은 Rust `npu-ir` 백엔드지만 Python 진입점이 둘이고, 서로 안 이어진다.

**World-1** `CompileModule.from_module → ir.Edf → EdfModule`
- Python에서 raw-tensor로 **run + profile 되는 유일한 경로**. 우리가 지금까지 쓴 것 전부.
- config가 12-knob `Config` **객체**뿐. `partial_sum_policy` 없음. tactic_hint enum에 production이 쓰는 `ForLlmModelComputeBound`가 아예 없음 (`{Default, ForLlmModelDecode, ForLlmModelPrefill, PrefillLongSequence, ForMisc, ForVision, NoConstraint}`만).
- dict 넘기면 `'dict' cannot be converted to 'Config'`. `set_compiler_config`(process-wide default)도 같은 Config.
- → **production config를 표현할 수 없음.** 접근 가능한 최선 hint(`ForLlmModelPrefill`)는 오히려 더 느려짐(gate S=2048: 2457us → 2864us).

**World-2** `furiosa.native_common.compiler.compile(model, (x,), config=<full 100-knob dict>, target_ir="edf")`
- full config(`partial_sum_policy: Bf16ForSplitAndChipAndCluster`, `tactic_hint: {ForLlmModelComputeBound: 100}`, `lowering_mode: Optimal`, `scheduler_beam_search` ...)를 받아 **컴파일은 됨** (`is_edf()=True`).
- **그러나 결과물(`CompiledGraph`)을 실행할 수가 없다.**

## 3. World-2 CompiledGraph는 왜 실행 못 하나 — 모든 경로 확인

`CompiledGraph` 메서드 8개(`serialize/deserialize/is_edf/is_pre_command_gen/encapsulate/task_num/cpu_task_num/npu_task_num`)에 `run`/`execute`가 없다.

- `serialize()` = 117MB 컨테이너, `ir.Edf` serde와 포맷이 달라 `ir.Edf.deserialize`가 `"missing field data"`로 거부. `encapsulate()` = 불투명 PyCapsule.
- **Route 1 (숨은 per-blob 실행 API): 없음.** 전체 live native 표면 introspect 결과 — `native_runtime`은 `NativeLLMEngine`(artifact_path만 받는 통째-LLM 엔진)뿐. tensor-I/O 실행자는 전부 World-1 타입(`EdfExecutor(ir.Edf)`, `DfgExecutor`, `Device.execute(Program)`, `EdfModule`). **CompiledGraph를 받는 것이 하나도 없음.**
- **Route 2 (비-LLM tensor 런타임): 없음.** `native_common`(full-config 컴파일러)를 쓰는 곳은 100% `furiosa_llm` 내부(builder/converter/pipeline). `furiosa.models`(모델 zoo)는 native_common도 CompileModule도 안 씀(아키텍처 정의일 뿐). **CompiledGraph는 furiosa_llm LLM 런타임에 용접되어 있고, 그 런타임은 tokenizer/kv-cache/sampling이 딸린 언어모델만 실행**한다.
- Route 2a(가짜 최소 LLM 아티팩트 위조) / Route 3(블롭 serde 파싱+device 드라이버 재구현)만 남는데 hacky/undocumented/고비용이라 후순위.

**★n-gram(2~3 커널 묶음)도 정확히 같은 벽★** — 묶어도 결국 CompileModule 경로라 World-1 config뿐. "우리 n-gram을 production 품질로 컴파일"을 SDK가 안 열어준다. 방법론(문맥/융합 포착)은 맞았지만, 재료의 fidelity가 막힌 것.

## 4. 같은 커널인가, 다른 커널인가 — 바이트 단위로 증명

"production은 다른 커널을 부르는 건가, 같은 커널을 다르게 컴파일하는 건가?" 를 확인.

동일 weight의 gate 행렬곱을 full config vs default config로 컴파일해 파이프라인 단계별 md5 비교:

```text
dfg → primitive → kernelized → prelower │ postlower → tiledgraph → lir → resourcelir → edf
└──────── 바이트 단위 IDENTICAL ────────┘ └──────────── DIFFERENT ────────────┘
        [연산 = 같은 커널]              ↑          [config가 tiling/tactic/스케줄 결정]
                                   갈라지는 지점 (postlower)
```

- `dfg/primitive/kernelized/prelower`: **바이트 단위 완전 동일** → 호출하는 커널(연산)은 100% 같다.
- `postlower`부터 갈라짐 → 여기가 config(`tactic_hint`, `partial_sum_policy`)를 보고 타일링·tactic·스케줄·TP리덕션을 정하는 지점.
- 생성 머신코드(`code_0.c`)도 다름: **115KB(full) vs 132KB(default)**.

결론: **`gcc -O0` vs `-O3`와 정확히 같다.** 같은 소스(커널), config에 따라 다른 어셈블리(머신코드), 19배 속도차. 실제 모델에선 여기에 더해 이웃과 **융합되어 Tokenwise supertask**가 되지만(커널 경계가 달라짐), matmul 연산 자체는 동일하다.

## 5. ratio-bridge 시도와 기각

"격리 측정을 못 고치면, World-1 값 × 보정계수 k로 production을 예측하면 되지 않나?" (k가 상수라면)

1B(TP=8, S=128 청크)에서 실측 — World-1 격리 / production supertask 비율:

```text
unit                     gap (world1/prod)
Tokenwise 블록            2.0x
lm_head (단일 matmul)     0.9x   ← World-1이 오히려 빠름
Attention core            1.6x
gate @ S=2048             ~19x
→ k는 상수가 아님 (op·shape 따라 1x ~ 22x 요동). 기각.
```

## 6. 왜 요동치나 — roofline이 gap을 지배

gap은 무작위가 아니라 **그 연산이 memory-bound냐 compute-bound냐**를 따라간다.

- 메모리-bound (lm_head 거대 weight/작은 토큰, 또는 작은 S에서 weight-load 지배): 두 세계 다 HBM 대역폭에 묶여 → gap ~1.
- compute-bound (큰 S matmul): World-1의 5% 가동률이 드러나 → gap ~19.
- production은 어느 영역이든 **물리 floor의 ~2배 안쪽**에 붙어감:

```text
8B prefill S=2048:
  matmul compute-floor (100% MFU)  = 15.4 ms   ← 물리 하한
  실제 prefill                      = 28.1 ms   ← floor의 1.8배
  World-1 matmul 합                 = 347  ms   ← floor의 22.5배
```

## 7. 남은 방향 — roofline 해석 예측 (미검증)

측정이 막혔으니 **물리로 예측**한다:

```text
op_time ≈ max( FLOPs/peak_compute , bytes/HBM_BW ) × eff
e2e     = Σ_supertask( 블록 roofline )
          within-block = 융합/겹침 → max(전체 compute, 전체 memory)
          between-block = 데이터 의존 → sum
```

- roofline(=max)은 shape에서 공짜로 나오는 물리 하한. `max()`가 memory/compute 영역 전환을 자동 처리.
- eff = 실행효율 잔차(겹침·배열점유율·오버헤드). 이게 World-1이 틀리고 production이 맞추는 어려운 부분. 단 production은 잘 최적화돼 늘 floor 근처라 eff가 **1.05~1.8 (~1.7배)**로 좁다 (World-1 gap의 22배 요동보다 훨씬 얌전).
- **이 방향의 생사 = eff가 op타입·영역별로 안정적으로 calibrate·일반화되냐.** 주현은 eff가 동적일 것이라 회의적이고, 타당하다. 미검증.

주의: eff는 격리 실행 없이 **B 레시피(프로파일 켠 실모델 재빌드+dump)로 레퍼런스 몇 config를 측정**해서 op타입·영역별로 뽑아야 한다 (measure-few-predict-many). 측정이 0은 아님.

## 8. B 레시피 — production supertask 실측은 된다

실제 모델을 재빌드+dump하면 supertask 타이밍은 얻을 수 있다 (bottom-up 예측은 아니지만 검증용 ground truth).

```text
빌드: ArtifactBuilder(model, compiler_config=CompilerConfig(
        compiler_config_overrides={"enable_tuc_profile": True, "profile_sync": True})).build(...)
실행: EDF_PROFILER_OUTPUT_PATH=<파일>  TUC_PROFILE_LEVEL=info  RUST_LOG=info,span::tuc=info  python run_profiled_dump.py
```

- ⚠️ `EDF_PROFILER_OUTPUT_PATH`는 디렉토리 아니라 **파일**. `FURIOSA_PROFILER_OUTPUT_PATHS`(dir)는 안 뱉음. `ENABLE_PERT_PROFILE`/`NPU_NONBLOCK`은 device 열기 깨뜨림.
- **granularity = SUPERTASK(융합 블록)**, 개별 matmul 아님. production은 QKV+gate/up/down을 `Tokenwise` supertask로, attention core를 `Attention`으로 융합. "gate matmul 하나의 latency"는 실제 모델에 존재하지 않는 단위. **즉 supertask = 융합된 n-gram** — 주현의 직감(문맥으로 융합 포착)이 NPU 실제 동작과 일치한다.
- 1B 실측(`~/llama1b_trace/edf.json`, meta-llama/Llama-3.2-1B-Instruct, TP=8, S=128 청크): Tokenwise@128 ~362K cyc(~327us), Attention@128 ~41K(kv=0), Attention@256 ~86K(kv=128), lm_head 1.28M cyc.
- ⚠️ furiosa-ai/Llama-8B는 raw 가중치가 없어(params-*만) 8B 재빌드 불가. 원본 meta-llama/Llama-3.1-8B(gated) 필요.

## 10. 결론과 다음

**한 줄 결론:** bottom-up-by-측정은 이 NPU에서 막혔다. NPU에서 커널 latency는 컴파일 문맥의 속성이라, 격리해서 재면 다른 물건이 된다 (GPU는 커널 라이브러리 고정이라 이 문제가 없다 — NPU 특유의 현상이고, 이것 자체가 finding이다). 방법론(n-gram/조합)은 맞았고, 재료의 fidelity만 죽었다.

다음에 볼 것:

- roofline × eff 예측기를 실제로 코드로 짜서, 1B supertask로 eff calibrate → 8B e2e(28.1ms)로 검증. eff가 안정적이면 이 방향이 산다.
- eff를 단일 상수로 볼지, op타입·roofline영역별 함수로 볼지.
- "NPU에서 조합의 원자 단위 = 연산자가 아니라 컴파일러 융합 단위(supertask)" 를 논문 결과로 정식화할지.
