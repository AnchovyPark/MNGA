## 0. 오늘의 목표

07-18에 "컴파일러가 자기 cost-model 예상치를 dump한다"를 찾았다([[2026-07-18]]). 근데 그 예상치를 검증한 건 **standalone 뭉치**(우리가 손수 만든 것)라 실측과 ~2.1~2.8x 차이났고 비율이 seq 따라 요동쳤다(못 믿음). 오늘은 **실제 모델 안에서(in-model)** 예상치를 뽑아 실측과 대조 → 컴파일러 예상치가 신뢰할 만한지 결판.

## 1. 실모델 dump 방법 (build_with_dump.py)

`FURIOSA_COMPILE_DUMP_PATH=<dir>` env를 켜고 모델을 빌드하면, 각 supertask 컴파일 시 `{tag}.summary/summary.json`을 dump한다 (tag = hash+{batch}_{input}_{attention}).

- 컴파일 캐시(`~/.cache/furiosa/llm/compiled_graphs`)를 우회해야 재컴파일+dump 됨. 1B는 param/graphmodule만 심링크한 fresh cache로 우회.
- dump 하나가 519MB(intermediate에 weight 임베드) → cleaner(`find -size +2M -delete` 루프)로 큰 파일 실시간 삭제, summary.json(2KB)만 유지. 디스크 안전.
- `enable_tuc_profile=True`로 빌드 → 결과 아티팩트가 프로파일 가능 = 실측과 같은 컴파일 조건.

## 2. 발견 — 컴파일러의 진짜 supertask 분해

런타임 trace는 "Tokenwise / Attention" 2개로 보이지만, 컴파일러 dump는 **더 잘게, 역할별로** 쪼갠다:

```text
supertask A: {attn_Q_proj, attn_K_proj, attn_V_proj, einsum_rope}   ← QKV 투영 + RoPE
supertask B: {attn_O_proj, mlp_gate_proj, mlp_up_proj, mlp_down_proj} ← O투영 + MLP
supertask C: {matmul_qk, softmax, matmul_score_v, k/v_scatter}       ← attention core
```

- op가 generic "mm"이 아니라 **역할명**으로 나옴 (실제 모델이라 각 matmul이 뭔지 라벨링됨).
- 런타임 "Tokenwise" = A + B 합. "Attention" = C.
- 아티팩트 33스테이지·164블롭 = 스테이지당 ~5블롭.
- 작은 seq(128): matmul_qk=0, softmax=0 → compute 거의 0, IO+overhead 지배. 큰 seq(1024): matmul_qk·softmax 커짐.

## 3. 핵심 결과 — k_type 안정 (in-model 예상 vs 실측)

각 조각(A+B=Tokenwise, C=Attention)의 예상 cycle을 실측 supertask cycle과 대조:

```text
seq   TW예상    TW실측    비율   | attn예상  attn실측  비율
128   213165   368000   0.58x  | 16276    41000    0.40x
256   292547   494000   0.59x  | 25522    75000    0.34x
512   463447   827000   0.56x  | 67599    143000   0.47x
1024  801617   1429000  0.56x  | 190521   477000   0.40x
```

- **Tokenwise: 예상/실측 = 0.56~0.59 (seq 128~1024 내내 거의 상수)** → 실측 ≈ 예상 × **1.75**
- **Attention: ~0.40 (거의 상수)** → 실측 ≈ 예상 × **2.5**
- **standalone(검증1)은 비율이 2.14→2.78로 요동쳤는데, in-model은 상수.** 맞는 context+grouping으로 뽑으니 컴파일러 예상치가 실측을 안정적으로 추적.
- 해석: 컴파일러가 **일정하게 낙관적**으로 예상(정적 cost 모델이 놓치는 hw 오버헤드가 ~상수 비율로 더해짐).

**결론: 컴파일러의 공짜 예상치 × 타입별 상수(k_TW≈1.75, k_Attn≈2.5) = 실제 latency. 그 상수가 seq에 안정적임을 1B에서 확인.**

## 4. weight 재확인 (중요)

- `furiosa-ai/Llama-*`는 **raw HF weight를 안 준다** — furiosa 샤딩 params-*(`params-...safetensors/model-0000N` = 디렉토리 안 조각) + prebuilt 아티팩트만. 그래서 furiosa-ai/Llama-8B로는 재빌드 불가(빌더가 raw safetensors를 못 찾음).
- raw는 **meta-llama org**: 1B = `meta-llama/Llama-3.2-1B-Instruct`(우리가 씀), 8B = `meta-llama/Llama-3.1-8B-Instruct`(gated, 접근권 확인됨).
- **latency는 weight 값이 아니라 shape+컴파일 config로 정해진다** — 샤딩(TP 배치)은 config(TP=8)가 결정, weight 값은 timing 무관. 그래서 우리 raw 빌드 latency ≈ furiosa official. (caveat: enable_tuc_profile=True 오버헤드로 절대값만 미세하게 높을 수 있으나, k_type 비율은 예상·실측 같은 빌드라 무관.)

## 5. 진행 중 / 다음

- **8B 일반화 검증 진행 중**: k_type(1.75, 2.5)이 8B에서도 같은가. `meta-llama/Llama-3.1-8B-Instruct`(raw ~16GB) 다운 + dump/profile 빌드(~2-3h) → 같은 대조. (furiosa-ai 8B로는 불가, raw 없음.)
- 8B 결과 나오면: k_type이 모델 크기 간 일반화되면 → 컴파일러 예상치 기반 예측기 성립. 다르면 → 모델별 calibration 필요.
