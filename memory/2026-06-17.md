## 1. 연구 방향 확정
- **성능 "개선"** 으로 결정 (예측 아님). 단순 KV 양자화는 제외.
- 주목점: RNGD SRAM. → Direction 2(hot KV 상주) vs Direction 3(SRAM streaming/tiling buffer).

## 2. RNGD 실제 스펙 (웹 확인, 핵심)
- **SRAM 256MB @ 384 TB/s**, HBM3 48GB @ **1.5 TB/s**, BF16 256 / FP8 512 TFLOPS, TCP, 1.0GHz.
- **핵심: SRAM의 무기는 용량이 아니라 대역폭(HBM의 256×).**

## 3. 핵심 측정 #1 — batch=8 decode TBT vs context (Llama-3.1-8B)
- prebuilt 아티팩트가 batch=8 decode bucket을 1k~32k까지 이미 보유 → **컴파일 없이** sweep.
- 결과(`~/lens_profiling_results/RNGD/Llama-3.1-8B-Instruct/bs8_ctx_sweep/`, profile=`profile_bs8_ctx_sweep.csv`):

| ctx | TBT(ms/tok) | 처리량 | KV(GB) |
|---:|---:|---:|---:|
| 800 | 16.5 | 486 tok/s | 0.8 |
| 3,800 | 19.5 | 410 | 3.7 |
| 7,800 | 24.6 | 325 | 7.6 |
| 15,700 | **33.5** | **239** | 15.3 |

- 분해: **weight 바닥 ≈15.6ms (유효 BW ~0.9 TB/s = peak의 64%)**, KV항 = 0.00114·ctx.
- **16k에서 KV가 decode 시간의 54%** (weight=KV 교차점 ctx≈14k). → **긴 context decode 병목이 KV cache HBM 트래픽임을 데이터로 확정 = 개선 표적.**
- 메모리 한계: `batch×context`가 HBM에 묶임. bs8≈28k / bs4≈57k / bs1≈230k 토큰이 상한.
  bs8×32k는 OOM(KV 32GB+weight 16GB>47.5). 로드 시 `--max-io-memory-mb 8192 --max-processing-samples 8`.

## 4. 두 방향 분석 (스펙 대입)
- **Direction 2 (hot KV 상주) — 용량의 벽**: KV 128KB/token. SRAM 256MB=총 2048토큰=bs8에서 256토큰/seq.
  16k면 KV의 1.6%만 상주 → 단순 최근-N 상주는 효과 ~0. **prefix 공유(배치가 같은 prefix 공유) 또는 sparse attention** 전제일 때만 의미. (sparse는 정확도 trade-off)
- **Direction 3 (SRAM = streaming/tiling buffer) — 스펙이 가리키는 답**: HBM 읽기 효율을 peak로.
  유효 0.9 vs peak 1.5 TB/s 격차(36%)가 헤드룸. **무손실, 전 구간 적용, 천장 ~1.56×.**
- **판정: Direction 3 주력, Direction 2는 prefix/sparse 니치 조건부.**

## 5. SDK 제어/관측 가능성 감사 (서브에이전트)
**제어 가능한 레버:**
- `USE_WTL_BACKEND=1` — attention 백엔드 LTW(기본,deprecated)↔WTL 전환 (on-chip 버퍼 전략 다름). import 전 설정.
- `chunk_size`(chunked attention 타일, 기본 8192) — attention layer별.
- `max_io_memory_mb`, `paged_attention_num_blocks`, `spare_blocks_ratio`, prefix cache.
- ⚠ paged `block_size`는 **1로 하드코딩**.
- ⚠ WTL/chunk_size는 **컴파일 시 baked** → A/B하려면 재컴파일 (1B는 저렴).
- 저수준: `furiosa/native_torch/memory.pyi`에 `_set_sram_mode/_set_memory_type/dram_alloc` 존재.

**커널 레벨 (가장 중요):**
- **TacticKernel** — 선언형 커널 작성(`furiosa.torch.TacticKernelModule(dsl_or_yaml)`), Virtual ISA.
  YAML 텐서마다 **`buffer_type: Sram|Dram`** 명시 = **SRAM 배치를 직접 제어하는 실체.**
- DSL 예시: `prng.py`(xoshiro, box-muller) — `#tactic_kernel_dsl`, `tk.Interleaving/Elementwise`, `ve.exec(...)`.
- 컴파일 경로: `TacticKernelModule(spec)` → `CompileModule.from_module(m, (cpu_inputs,))` → device.

**관측 (kernel profiler):**
- `CompileModule.from_exported(ep)` → `generate_profiles` → `edf.npu_node.build_tuc_profile_spans`.
- span 종류: Task / TuExec(연산) / DMA(전송) / ParallelCopy / StoTrf. **단위는 시간(us)뿐, byte 카운트 없음.**
- → compute vs transfer 분해 가능, SRAM 점유율/hit rate 직접 카운터는 없음(바이트는 모델에서 계산해 곱함).

## 6. 핵심 측정 #2 — Lv0 decode attention 분해 (`attn_decode_profile.py`)
결과(`attn_decode_profile_results.csv`, MHA H=32 D=128, q_len=1):

| ctx | KV(MB) | DMA us | TuExec us | task us | 유효BW | %peak |
|---:|---:|---:|---:|---:|---:|---:|
| 1k | 16.8 | 87 | 12.9 | 91 | 0.19 | 13% |
| 8k | 134 | 985 | 90.8 | 940 | 0.14 | 9% |
| 16k | 268 | 1878 | 182 | 1882 | 0.14 | 10% |

- **DMA : 연산 ≈ 10 : 1** → decode attention은 압도적 HBM-전송 바운드.
- DMA가 context에 **정확히 선형** (더블링마다 ×2).
- **task_us ≈ DMA_us** → DMA 사실상 직렬, 병렬화 안 됨 = streaming 비효율의 직접 증거.
- ⚠ isolated micro-op이라 %peak가 10%로 낮게 나옴(invocation/GEMV 오버헤드). full-model 64%가 대표 베이스라인. → **헤드룸이 attention KV-read에 집중.**

## 7. 핵심 검증 — TacticKernel 스파이크 (`tactic_kernel_spike.py`)
- (1) 커스텀 DSL 커널(elementwise add) CPU 실행: **PASS**
- (2) 같은 커널 **RNGD 컴파일 + device 실행: PASS** ← 새 커널이 칩에서 정확 동작.
- (3) `buffer_type: Sram`(moe YAML, SRAM 텐서 6개) **파싱 OK** — SRAM 명세 포맷 유효.
  - 미완: SRAM-explicit 커널 device 실행. 막힌 이유는 (a) `CompileModule.from_module` 인자 사용 버그(kwargs 4번째 positional로 넘김), (b) moe 커널이 FilterCompaction이라 유효 입력 필요. **기능 한계 아님.**
- **결론: RNGD에서 kernel-level 작성·컴파일·실행 가능 + SRAM 명세 유효 → Direction 3/2의 진짜 커널 연구 실현가능.**
