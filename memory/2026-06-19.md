## 2. 프로파일러 — 정확히 무엇을 어떻게 뱉나 (오늘 규명, 방법론 핵심)
- 공식 `RNGDProfiler`(chrome-trace)는 torch 2.10+xpu / Intel PTI 충돌로 **사용 불가** → 저수준 경로:
  `CompileModule.from_exported(ep)` → `cm.generate_profiles(dev)` → `cm(.., profiles=..)` →
  `cm.edf.npu_node.build_tuc_profile_spans(pc, di, ns)` → `FunctionEvent` 리스트.
- **span 3종**: `Renegade::TuExec`(연산 명령들), `DMA`(전송 명령들), `Task`(전체 실행 1개 래퍼).
  각 span = name + time_range.start/end/elapsed_us + device_index + device_resource_id.
  JSON(chrome-trace)으로 직렬화 가능. byte 카운터는 없음(모델에서 계산).
- **★커널 시간 = Task** (device latency, 겹침 반영). **합(sum) 쓰면 안 됨**:
  한 커널이 ~200개 TuExec 명령으로 쪼개져 **병렬 레인에서 겹쳐** 돎 → sum > Task.
  올바른 값 = **union**(겹침 병합). 예(matmul): sum(TuExec)=5925 vs union=4857 ≈ Task=4885.
- **겹침/idle 전부 계산 가능**: 연산↔DMA 겹침(예 1822us), 연산 내부 병렬도 sum/union=1.22×.
  compute-bound 예(matmul): 연산 union≈Task, DMA(1851)는 거의 다 연산 뒤에 숨음, idle≈0.
- **엔진**: 컴파일러 IR엔 `ein_ops`(DPE=행렬엔진) vs `vector_ops`(VE=벡터엔진, VRF)가 분리돼 있고
  플래그(`separate_vector_ops_from_dpe`, `hide_qk_einsum_under_ve`)로 동시 실행/오버랩.
  **그러나 profiler span은 둘 다 `TuExec` 한 이름으로 뭉침** → DPE/VE 시간 분리는 span 이름으론 불가
  (backtrace/ProfileLevel로 가능성 있음, 미확인).
- **★TP=1 (단일 PE)★**: `CompileModule.from_exported`는 **단일 PE 컴파일**. 모든 span device_index=0/`rngd:0`.
  유효 BW가 죄다 0.19 TB/s = **단일 PE 대역폭의 ~100%** (peak 1.5는 8-PE 합산).
  → [[2026-06-17]]·§A의 "13% peak, 헤드룸 크다"는 **1-PE를 8-PE peak에 비교한 착시.** 실제론 1 PE를 꽉 씀.
  실제 serving은 **TP=8** (weight·scores가 8 PE로 샤딩). 다음 과제.

## 3. ★합성(fusion) 측정 — 메인 결과★
**Formal 프레임**: 분석 단위 = 그래프의 방향 엣지 **A→B**(커널 두 개 연결, LLM 의미 무관, matmul/vector 종류만).
척도 = **Task**. 측정 = `cost(A)`, `cost(B)`, `cost(A→B fused)`.
- iso = 각 커널 단독 그래프 컴파일. **fused = A,B를 한 그래프로 joint-compile** (컴파일러가 엣지 가로질러 fusion).
- saving = `A + B − fused` (절대 us). 비율은 안 씀. **관심 = "어느 커널이 흡수돼 사라졌나".**

**중간 정정**: 처음 6-op 체인 실험(`kernel_composition.py`)에서 본 decode "합성격차 2.6×"는
host **wall**로 잰 dispatch 오버헤드였음. **Task로 재니 사라짐** → Task가 정답.

### TP=1 인접쌍 결과 (`pair_fusion.py`)
**decode (B=8, S=1, ctx=2048)** — vector가 전부 흡수, saving 작음(vector가 원래 3~18us라 흡수할 게 적음):
| edge | A | B | fused | saving |
|---|--:|--:|--:|--:|
| rmsnorm→matmul | 3 | 608 | 608 | +3 |
| matmul→silu | 608 | 3 | 608 | +3 |
| silu→mul | 3 | 5 | 5 | +3 |
| mul→matmul | 5 | 604 | 609 | ~0 |
| matmul→add | 604 | 3 | 604 | +3 |
| matmul→softmax | 974 | 18 | 981 | +11 |
| softmax→matmul | 18 | 965 | 965 | +18 |

**prefill (B=1, S=2048)** — vector가 1000us대라 흡수가 크게 보임; **attention은 anti-fusion**:
| edge | A | B | fused | saving | 결과 |
|---|--:|--:|--:|--:|---|
| rmsnorm→matmul | 172 | 10069 | 10116 | +123 | vector 흡수 |
| matmul→silu | 10067 | 523 | 10104 | +486 | vector 흡수 |
| silu→mul | 523 | 800 | 793 | +530 | vector 흡수 |
| mul→matmul | 800 | 8677 | 8910 | +566 | vector 흡수 |
| matmul→add | 8676 | 251 | 8714 | +213 | vector 흡수 |
| **matmul→softmax (QK)** | 1540 | 2663 | 6352 | **−2149** | **anti** |
| **softmax→matmul (AV)** | 2664 | 1626 | 7573 | **−3283** | **anti** |

### 결론
1. **fused ≈ max(A,B): 붙이면 "vector 커널"이 옆 matmul에 흡수돼 사라지고, matmul 비용은 그대로.**
   saving = 흡수된 vector의 단독 비용.
2. **연결 텐서가 SRAM을 넘으면 부호가 뒤집힘(anti-fusion).** attention scores/probs `[B,NH,S,S]`가 그 텐서:
   - B=8/S=512 = 134MB: QK 정상 fusion, AV만 anti.
   - **B=1/S=2048 = 268MB: QK·AV 둘 다 anti.** → **scores 크기가 부호를 가르는 변수.**
   - anti 정체 다름: **AV = DMA 폭증(probs 재스트리밍)**, **QK = DMA 그대로인데 Task↑(연산/스케줄).**
3. decode는 vector가 애초에 작아 "fusion 효과 없음"이 아니라 **흡수할 게 없음**(가산적).

## 4. furiosa-llm attention 사실관계 (오늘 소스 확인)
- attention은 **eager**로 작성: `ops/scaled_dot_product.py` → `q@kᵀ → softmax → p@v`. flash 단일 op 없음.
- `chunked_scaled_dot_product_attention`(chunk_size 기본 8192) 존재하나 **opt-in**:
  `use_chunk_attention = chunk_size is not None and is_prefill`, 그리고 `query_len > 8192`일 때만.
  **standard Llama는 chunk_size를 set 안 함 → prefill도 vanilla full attention.** (chunked는 block-diagonal이라 일반 경로 아님)
- decode = vanilla지만 q_len=1이라 scores가 작아 anti-fusion 무관.
- 컴파일러엔 전용 `AttentionKernel`/`MegaAttentionKernel`+`use_attention_kernel` 존재 →
  **eager 패턴을 특수 커널로 lower할 가능성** (Python eager ≠ HW eager일 수 있음, 미확인).
- → 내 pair 실험의 attention anti-fusion은 "전용 attention 커널/chunking을 안 쓴 naive 합성"의 결과
  = **왜 그런 메커니즘이 필요한지**를 보여주는 negative control.

## 7. TP=8 / all-reduce 조사 — ★막힘 확인 (오늘 결론)★
연구 관심: **TP에서 all-reduce(집합통신)가 포함되면 latency가 얼마나 늘어나는가.** (단순 sharding 1/8 프록시는 통신을 빼버려 무의미 — 사용자 명확히 함.)

조사·실험으로 확인한 막힘:
- **단일 op은 8 PE 자동 매핑 안 됨.** `CompileModule.from_exported`=1 PE. generic compiler config(`create_default_compiler_config`)엔 **num_pe 키 없음**. num_pe를 담는 config 생성기(`create_llm_compiler_config_with_layer_range`)는 **model preset-locked** — model_type/task/params 조합 100+개 다 넣어도 `find_compiler_config` 0개 반환.
- **손수 MpppConfig로 all_reduce 꽂기도 막힘**: `ModelRewriter(gm, cfg).rewrite()` → `ShardingPropagator: only supports cases where all inputs are replicated` (matmul). **샤딩된 matmul 입력을 전파기가 구현 안 함** → bare op TP 불가. SDK TP는 **인식된 모델 패턴 전용.**
- device 포맷 `rngd:칩:PE수`(PE수∈{1,2,4,8}) 발견 — 칩 내 8 PE 타일링 시 부분합 cluster-reduce(=칩 내 all-reduce)가 생기나, 이걸 bare op에 트리거하는 generic 경로가 없음.

**all-reduce span 분리 가능성 (feasibility)**:
- 저수준 span: op 의미 라벨 없음(종류 이름 `TuExec`/`DMA`/`Task`만). all_reduce는 DMA로 떨어져 HBM-DMA와 안 갈림. inter-PE 통신이 별도 span 종류(`TensorDmaClusterSwap`/`ReduceScatter`/`ClusterSync`)로 찍히면 분리 가능하나 **단일 PE라 미확인.**
- 파이프라인 레벨: collective가 **이름 붙은 supertask**(all_reduce/all_gather)라 분리 가능하나, 그 프로파일러=`RNGDProfiler`가 **PTI 충돌로 못 씀.**
- → all_reduce 분리 측정은 **장애물 3겹**(span 라벨 없음 / 파이프라인 프로파일러 PTI 충돌 / 진짜 TP=8 모델 빌드 필요)으로 고비용·불확실.

**미해결 결론**: TP=8 / all-reduce latency는 현재 micro-benchmark 틀로 측정 불가. 추후 옵션:
- (a) 진짜 TP=8 모델 빌드 + 파이프라인 프로파일 훅 뚫기(PTI 우회) — 고비용.
- (b) **차분 측정**: comm 있는 구성 vs 없는 구성 Task 차이 = all-reduce 기여(라벨 불필요) — 가성비 최선.
- (c) all-reduce 접고 single-PE 합성/예측 결과로 진행.
